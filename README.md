
# XAI for Arabic Legal RAG

This framework is an Arabic legal adaptation of a perturbation-based explainability pipeline inspired by RAG-Ex. Its purpose is to explain which parts of the retrieved legal context most strongly influence the final answer generated by a language model.

The framework is designed for Retrieval-Augmented Generation (RAG) settings, especially in legal question answering, where it is important not only to produce an answer, but also to understand whether that answer is actually grounded in the retrieved legal evidence.

Rather than trying to inspect the internal weights, activations, or hidden states of the model, this framework explains the model externally by systematically modifying the input evidence and observing how the output changes.

**The key idea behind the framework is that it does not attempt to explain the model by inspecting its internal parameters. Instead, it explains the model externally by observing how the output changes when the supporting evidence changes. This is the core principle behind the system you built.**

---

## 1. Overview

At a high level, the framework works through five main components:

- The **tokenizer** breaks the input text into smaller explainable units, such as sentences.
- The **perturber** modifies one unit at a time to create alternative versions of the input.
- The **generator** runs the language model again on each modified version.
- The **comparator** measures how much the new answer differs from the original answer.
- The **explainer** connects all of these steps into a single workflow and ranks the input units by importance.

In short, the framework works by taking a legal context, perturbing one clause at a time, regenerating the answer, comparing the new answer to the original one, and then ranking the clauses according to how much they influenced the output.

This means the framework performs explanation through **behavioral sensitivity analysis**. If changing one legal clause causes a large change in the generated answer, that clause is considered highly important. If changing it has little or no effect, it is considered less influential.

---

## 2. Why this framework exists

In legal RAG systems, grounding matters. A model may generate a fluent and convincing answer, but fluency alone does not guarantee that the answer is actually based on the retrieved legal evidence. This framework helps address that issue by showing which parts of the context were influential in producing the answer.

This makes the system useful for several goals:

- verifying grounding in retrieved legal text
- identifying influential legal clauses
- analyzing model sensitivity to factual and semantic changes
- detecting whether the model relies on wording or meaning
- improving trust and transparency in Arabic legal QA systems

The framework is especially useful in hallucination-reduction studies, because it provides a way to inspect whether the answer changes when the legal evidence changes.

## 3. Folder-to-method mapping

The implementation is organized into a root framework directory named `ragex_framework`, which contains the following main components.

### `dto.py`

This file defines the core data structures used across the framework. It includes:

- `ExplanationGranularity`, which specifies the level of analysis such as word, phrase, sentence, or paragraph.
- `FeatureImportance`, which stores a feature and its corresponding importance score.
- `ExplanationDto`, which packages the final explanation results, including the ranked features and the original input.

This file serves as the output schema of the framework.

---

### `utils.py`

This file provides utility functions for score handling. It includes:

- normalization of raw similarity values into the range `[0,1]`
- reversal of similarity scores into dissimilarity-based importance scores
- sorting of features according to descending importance

These functions ensure that explanation results are standardized and easy to interpret.

---

### `explainer/`

This folder contains the orchestration logic of the framework.

- `generic_explainer.py` defines the abstract pipeline. Its `explain()` method performs the full explanation workflow by calling:
  - `get_features()`
  - `get_perturbations()`
  - `get_post_perturbation_results()`
  - `get_comparator_scores()`

- `generic_generator_explainer.py` provides the concrete implementation for generator-based language models. It connects the tokenizer, perturber, generator function, and comparator into a working explanation system.

This folder acts as the central controller of the explanation process.

---

### `modules/tokenizer/`

This folder handles feature extraction from the input text.

- `arabic_legal_tokenizer.py` implements an Arabic legal tokenizer that supports sentence-level, word-level, phrase-level, and paragraph-level segmentation.

In our implementation, sentence-level segmentation is the primary setting because legal clauses are naturally expressed as sentence-like units. The tokenizer therefore converts the retrieved legal context into a list of candidate features for explanation.

---

### `modules/perturber/`

This folder generates modified versions of the input text.

- `leave_one_out_perturber.py` removes one feature at a time, asking what happens if a given sentence is absent.
- `reorder_perturber.py` changes the local order of words within a feature to test structural sensitivity.
- `llm_based_perturber.py` uses a language model to produce controlled sentence-level perturbations. This includes:
  - `RandomNoisePerturber`
  - `EntityManipulationPerturber`
  - `AntonymInjectionPerturber`
  - `SynonymInjectionPerturber`

These perturbation strategies allow the framework to test different types of reliance, including omission, structural change, factual change, meaning reversal, and wording variation.

---

### `modules/comparator/`

This folder evaluates how much the generated answer changes after perturbation.

- `embedding_comparator.py` computes semantic similarity using sentence embeddings and cosine similarity.
- `generic_comparator.py` provides lexical comparators such as Levenshtein distance.
- `n_gram_overlap_comparator.py` measures phrase-level overlap using n-gram distance.
- `legal_hybrid_comparator.py` combines semantic, lexical, and structural signals into a weighted hybrid score.

This folder is responsible for transforming output differences into numerical importance scores.
